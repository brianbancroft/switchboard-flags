## Write Safety Model

All mutating tools require:

```json
{
  "approval": {
    "confirmed": true,
    "reason": "User explicitly approved this write in the current conversation."
  }
}
```

This is enforced in the MCP input schema and is part of the platform’s safety story.

## System Tools

| Tool | Description | Example prompt |
| --- | --- | --- |
| `get_switchboard_health` | Check whether the Hono API is alive | `Check whether Switchboard is up.` |
| `get_switchboard_database_health` | Check DB connectivity through the API boundary | `Is the Switchboard database reachable?` |

## Environment Tools

| Tool | Description | Example prompt |
| --- | --- | --- |
| `list_environments` | List environments visible to the configured user context | `List my Switchboard environments.` |
| `get_environment` | Get one environment by ID or exact name | `Show me the production environment details.` |
| `create_environment` | Create a new environment | `Create a nightly environment for QA.` |
| `update_environment` | Update environment metadata | `Rename nightly-eu to staging-eu.` |
| `delete_environment` | Delete an environment | `Delete the old demo environment after I confirm.` |

## Prompting Tips For Assistants

- Mention the environment by exact name whenever possible.
- Use real UUIDs for user-specific overrides.
- Specify whether you want a read-only inspection or an actual write.
- Include client context such as `clientLabel` or `url` when debugging evaluation.
